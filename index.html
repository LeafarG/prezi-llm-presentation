História

De um neurônio matemático publicado em 1943 a agentes autônomos que executam workflows completos em 2026. Uma linha do tempo das ideias que tornaram os LLMs possíveis.

1943

McCulloch-Pitts: o primeiro neurônio matemático

Warren McCulloch e Walter Pitts publicam "A Logical Calculus of the Ideas Immanent in Nervous Activity". Propõem um modelo simplificado de neurônio como uma porta lógica binária. É a base conceitual de tudo que veio depois — mas ainda sem aprendizado.

1958

Perceptron ROSENBLATT

Frank Rosenblatt cria o Perceptron no Cornell Aeronautical Laboratory: o primeiro algoritmo que aprende a partir de dados. Roda no Mark I, um computador do tamanho de uma sala. O New York Times publica em primeira página.

1969

Minsky & Papert — primeiro AI winter

Perceptrons (livro) demonstra matematicamente que o Perceptron simples não resolve o XOR. Cortes no financiamento. A pesquisa em redes neurais fica em geladeira por quase 20 anos.

1986

Backpropagation RUMELHART · HINTON · WILLIAMS

O algoritmo de retropropagação é popularizado. Redes com múltiplas camadas voltam à mesa — agora dá para treinar representações úteis. GPUs ainda estão uma década distante.

1997

LSTM HOCHREITER · SCHMIDHUBER

Long Short-Term Memory resolve o problema de dependências de longo prazo em sequências. Por 20 anos, LSTM + variantes são o estado da arte em tradução, reconhecimento de fala e modelagem de linguagem.

2012

AlexNet KRIZHEVSKY · SUTSKEVER · HINTON

CNN de 8 camadas e 60M de parâmetros arrasa a ImageNet: erro top-5 cai de 26% para 15%. Treinada em duas GTX 580. É o gatilho do deep learning moderno. A ImageNet moment.

2014

Seq2Seq e Bahdanau Attention

Sutskever et al. (Google) publicam o modelo Sequence-to-Sequence: encoder-decoder para tradução. No mesmo ano, Bahdanau, Cho & Bengio introduzem o mecanismo de atenção — a peça que vai destravar a próxima década.

2017

"Attention Is All You Need" VASWANI ET AL.

O paper da Google Brain que muda tudo. Propõe o Transformer: arquitetura baseada inteiramente em atenção, sem recorrência ou convolução. Treina em paralelo, escala melhor. 8 autores, 1 paper, 1 década de impacto. Fundação de todo LLM moderno.

2018

BERT e GPT-1

Google publica BERT (encoder-only, bidirecional) — vira a base da busca moderna. OpenAI publica GPT-1 (decoder-only, 117M de parâmetros) — primeira iteração da família que vai dominar o mundo.

2019

GPT-2 1.5B

OpenAI lança GPT-2 com 1.5B de parâmetros. Tão capaz de gerar texto coerente que decidem não liberar o modelo completo por medo de uso indevido (deepfakes de texto). Liberado gradualmente em estágios ao longo de 2019-2020 — caso emblemático de responsible disclosure.

2020

GPT-3 175B

175 bilhões de parâmetros. In-context learning (few-shot) emerge como capacidade nova sem treino adicional. A API da OpenAI vira plataforma. Fundação do GitHub Copilot, Notion AI e dezenas de startups. A era dos produtos começa.

2022 · 30/NOV

ChatGPT (GPT-3.5)

OpenAI lança ChatGPT baseado em GPT-3.5 com RLHF. 100 milhões de usuários em 2 meses — aplicativo de crescimento mais rápido da história. A IA sai do laboratório e vira conversa de família. A corrida começa.

2023

GPT-4, Claude, Llama, Gemini

GPT-4 (mar) traz multimodalidade. Claude 2 (jul) leva o contexto a 100k tokens. Llama 2 (jul) abre o código da Meta. Gemini 1.0 (dez) entra nativo-multimodal do Google. Preços caem 90% em 12 meses. API vira commoditie.

2024

Era do raciocínio: OpenAI o1

OpenAI lança o1 (set/2024): introduz inference-time compute scaling. O modelo gera chain-of-thought interno antes de responder. Em matemática e código complexo, supera GPT-4 sem ser maior. Reasoning vira categoria própria.

NOV · 2024

MCP — Model Context Protocol

Anthropic lança o MCP: padrão aberto para conectar LLMs a ferramentas e dados. "USB-C para IA." Adotado em meses por OpenAI, Google DeepMind e a maioria dos frameworks de agentes. Ecossistema explode em 2025.

2025

Agents everywhere

DeepSeek-R1 (jan) — reasoning open-source. Claude Code (fev) — terminal agent da Anthropic. Operator, Computer Use, Agents SDK. Agents começam a executar workflows completos: navegação web, edição de código, automação de planilha.

2026

Microsoft Cowork e a era agentic

Microsoft 365 Copilot Cowork atinge GA (jun/2026): agentes autônomos com Work IQ que executam tarefas de longa duração no ecossistema M365. Multi-agent orchestration vira commodity. Memória persistente entre sessões. IA deixa de ser ferramenta e vira colega de trabalho.

// 02

Tipos de LLM

Modelos diferentes para trabalhos diferentes. A escolha do tipo certo define custo, latência, qualidade e onde o modelo pode rodar.

S

SLM — Small Language Model

Menos de 10B de parâmetros. Cabe em laptop, celular ou Raspberry Pi. Latência sub-segundo, custo por requisição próximo de zero.

Phi-3 Gemma 2B Llama 3.2 1B/3B Qwen2.5 Ministral
Uso típico: edge devices, offline, privacidade
L

LLM — Large Language Model

10B a centenas de bilhões de parâmetros. Capacidade geral forte, multimodal, janelas de contexto longas. A maioria dos produtos comerciais roda nessa categoria.

GPT-4o Claude 3.5/4 Gemini 1.5/2.0 Llama 3.1 70B Mistral Large
Uso típico: produtos SaaS, APIs, chatbots
F

Frontier Model

O topo da pirâmide. Treinados com dezenas de milhões de dólares de compute. Capacidade máxima em raciocínio, código, ciência, multimodalidade. Geralmente fechados e caros.

GPT-5 Claude 4 Opus Gemini 2.5 Pro o1 / o3 DeepSeek-V3
Uso típico: pesquisa, problemas difíceis, agentes
R

Reasoning Model

Modelos treinados (ou pós-treinados) para raciocinar passo a passo antes de responder. Adicionam tokens de pensamento interno que não aparecem ao usuário.

o1 · o3 DeepSeek-R1 Claude extended thinking Gemini Thinking
Uso típico: matemática, código complexo, planejamento

Por pesos: aberto vs fechado

Open weights

Pesos disponíveis para download e self-host. Licenças variam: Apache 2.0 (Mistral), Llama Community (Meta), custom permissive (Qwen, DeepSeek). Permite auditoria, fine-tune e uso privado.

Llama Mistral Qwen DeepSeek Phi Gemma

Closed weights

Acesso apenas via API. Pesos nunca são liberados. Modelo de negócio baseado em uso. Mais capacidade bruta no curto prazo; menos controle e privacidade.

GPT (OpenAI) Claude (Anthropic) Gemini (Google) Grok (xAI)

Por arquitetura

E

Encoder-only

Lê e representa texto, não gera. Usado para classificação, embeddings, busca semântica, NER. BERT e derivados.

Famílias: BERT, RoBERTa, DeBERTa, E5, BGE
D

Decoder-only

Gera texto autoregressivamente. É a arquitetura dominante dos LLMs modernos. Tudo que responde com texto fluente está aqui.

Famílias: GPT, Claude, Llama, Mistral, Qwen, Gemini
ED

Encoder-decoder

Encoder lê a entrada, decoder gera a saída. Tradicionalmente forte em tradução e sumário. T5, BART e derivados modernos.

Famílias: T5, FLAN-T5, BART, mT5
// 03

Thinking vs No-thinking

A distinção mais importante em 2025-2026: a diferença entre um LLM que responde direto e um que pensa antes de responder.

Regra geral: tarefas que um humano resolveria "de cabeça" (chat, sumário, e-mail) não precisam de thinking. Tarefas que exigem planejamento, verificação ou múltiplas etapas (matemática, código complexo, estratégia) se beneficiam enormemente do thinking.

No-thinking direto

O modelo recebe o prompt e gera a resposta imediatamente, token a token. Padrão da maioria dos modelos até 2024. Ótimo para tarefas onde o caminho até a resposta é curto e o modelo tem confiança alta.

  • Latência: 100ms – 2s
  • Custo: $0.0001 – $0.01 por request
  • Throughput: alto, paralelizável
  • Boas para: chat, Q&A, sumarização, tradução, código simples

Modelos: GPT-4o · Claude 3.5/4 Sonnet · Gemini 1.5/2.0 Flash · Llama 3.1

Thinking raciocínio

O modelo gasta tokens internos (chain-of-thought) antes de produzir a resposta visível. Pensa, verifica, planeja, refaz. Em problemas difíceis a qualidade sobe muito — mas a latência e o custo explodem.

  • Latência: 5s – 60s+ (às vezes minutos)
  • Custo: $0.01 – $1+ por request
  • Throughput: baixo, sequencial
  • Boas para: matemática, código complexo, planejamento, pesquisa, agentes

Modelos: o1 · o3 · DeepSeek-R1 · Claude 4 Opus extended thinking · Gemini Thinking

Como funciona o thinking

O modelo gera uma sequência de tokens "privados" entre o prompt e a resposta final. Esses tokens não são mostrados ao usuário, mas são contabilizados no custo. O modelo pode:

  • Decompor o problema em sub-problemas
  • Tentar uma abordagem, verificar, refazer
  • Criticar a própria resposta antes de finalizar
  • Ramificar e comparar caminhos alternativos (em alguns modelos)

Inference-time compute scaling: a ideia de que mais "pensamento" (mais tokens de raciocínio) leva a respostas melhores. Foi o insight por trás do o1 e mudou a forma como pensamos sobre scaling — antes era só treino; agora também é inferência.

Quando usar cada

Cenário Recomendação
Chat de produto / atendimento No-thinking — latência importa mais que perfeição
Sumarizar um documento No-thinking — tarefa de extração
Resolver equação simbólica Thinking — precisa planejar passos
Gerar código de produção Thinking — vale a pena pensar antes de escrever
Planejar viagem multi-cidade Thinking — várias restrições interagem
Classificar sentimento No-thinking — tarefa simples e barata
Pesquisa jurídica com precedentes Thinking — raciocínio encadeado é crítico
Agente autônomo multi-step Thinking — verificação constante

Padrão recomendado em produção: tente primeiro com modelo no-thinking (rápido e barato). Se a resposta for insuficiente, escale para thinking. Cascata: thinking só quando necessário.

// 04

Frameworks: Chat, Agent, Harness

Três camadas conceituais que confundem muita gente. São padrões diferentes de uso, com limites e propósitos próprios.

Chat

Interface conversacional

Você pergunta, o modelo responde. Stateless (cada mensagem é independente, sem memória de longo prazo além do contexto da conversa). O foco é UX conversacional — digitar, ler, refinar.

Características: baixa latência, alta QPS, sem ferramentas, sem memória persistente, sem objetivo a longo prazo.

Exemplos: ChatGPT, Claude.ai, Gemini, Copilot Chat, Poe, LMSYS Chatbot Arena

Agent

Execução autônoma de tarefas

LLM + loop + ferramentas + memória + objetivo. O agente recebe uma tarefa, planeja, executa passos, observa resultados e itera até entregar um resultado final. O foco é completar uma tarefa, não conversar.

Características: loop de tool use, memória de trabalho, capacidade de replanejar, observabilidade de cada passo, tratamento de erro.

Exemplos: OpenAI Agents SDK, Anthropic Computer Use, Operator, Manus, Devin, AutoGPT, Browser Use

Harness

A camada que torna o modelo útil no dia a dia

Software que envolve o LLM dando capacidades persistentes: integração com IDE, sistema de arquivos, terminal, calendário, e-mail, codebase. É a "alavanca" que transforma um modelo genérico em ferramenta produtiva para um contexto específico.

Características: UX específica de domínio, integração profunda com sistema operacional/aplicações, permissões e segurança, human-in-the-loop.

Exemplos: Claude Code, GitHub Copilot, Microsoft 365 Copilot, Microsoft Cowork, Cursor, Cline, Continue

Como se relacionam: o chat é a UI. O agent é o padrão de execução. O harness é o produto que você usa todo dia.

Um harness pode conter agentes. Um agente pode ter uma interface de chat. Mas conceitualmente são camadas diferentes com objetivos diferentes.

Confusão comum: "Copilot" e "Cowork" são da Microsoft, mas GitHub Copilot é um harness de desenvolvimento (IDE), enquanto Microsoft 365 Copilot é um harness de produtividade (Office), e Microsoft Cowork é um agente autônomo que trabalha sobre o M365. Nomes parecidos, produtos diferentes.

// 05

Ferramentas (Tools)

LLMs são, no fundo, máquinas de texto. Tools dão "mãos" — acesso estruturado ao mundo externo. Sem tools, todo agente seria só um chatbot glorificado.

Conceitos fundamentais

1

Function calling

O modelo recebe um schema de funções disponíveis (nome, descrição, parâmetros JSON). Quando decide que uma função resolve a tarefa, retorna um objeto estruturado com o nome da função e os argumentos. Você executa a função do seu lado e devolve o resultado.

Providers: OpenAI · Anthropic · Google · Cohere · Mistral
2

Structured output

Garante que a saída do modelo segue um schema (JSON Schema, Pydantic, Zod, etc.). Usa constrained decoding ou grammar-guided generation para que o modelo não consiga produzir JSON inválido.

Implementações: JSON mode · tool_choice · grammar · guidance · outlines
3

Tool use loop

O ciclo clássico de um agente: modelo chama tool → recebe resultado → decide próximo passo → chama outra tool → ... → finalmente entrega resposta final. O harness implementa esse loop com limites (max steps, timeout, retry).

Limites típicos: 5–25 steps · timeout 30–120s · retry com backoff

Exemplo: function calling

O modelo recebe tools disponíveis e decide quais usar:

// 1. Definição da tool (enviada ao modelo)
{
  "type": "function",
  "function": {
    "name": "get_weather",
    "description": "Retorna o clima atual de uma cidade",
    "parameters": {
      "type": "object",
      "properties": {
        "city": { "type": "string", "description": "Nome da cidade" },
        "unit": { "type": "string", "enum": ["celsius", "fahrenheit"] }
      },
      "required": ["city"]
    }
  }
}

// 2. Resposta do modelo (decide chamar a tool)
{
  "tool_calls": [{
    "name": "get_weather",
    "arguments": { "city": "São Paulo", "unit": "celsius" }
  }]
}

// 3. Você executa e devolve o resultado
{ "tool_call_id": "...", "content": "{\"temp\": 18, \"sky\": \"nublado\"}" }

// 4. Modelo usa o resultado e responde
"Em São Paulo agora está 18°C com céu nublado."

Padrão ReAct

O padrão clássico de raciocínio + ação, formalizado em 2022 por Yao et al.:

Thought 1: O usuário quer saber o clima de SP. Action 1: get_weather(city="São Paulo") Observation 1: {"temp": 18, "sky": "nublado"} Thought 2: Tenho a informação. Formato a resposta. Action 2: finish(answer="Em SP agora está 18°C, nublado.") Observation 2: Task complete.

Tipos comuns de tools

Web search

Busca em tempo real na web. Implementado por Tavily, Brave Search, SerpAPI, Bing.

Code execution

Sandbox Python/JavaScript. Permite o modelo rodar código, instalar deps, gerar gráficos.

File system

Leitura, escrita, busca. Fundamental em coding agents (Claude Code, Cline).

Database queries

SQL ou vector search. Text-to-SQL é o subcampo clássico. RAG também conta.

API calls

REST/GraphQL genéricos. Stripe, GitHub, Slack, Gmail, Calendar — qualquer coisa.

Browser automation

Playwright, Puppeteer. Computer Use da Anthropic é o estado da arte.

// 06

MCP — Model Context Protocol

O padrão aberto criado pela Anthropic em novembro de 2024 que se tornou a espinha dorsal da integração entre LLMs e o mundo externo. USB-C para IA.

O que é

MCP é um protocolo aberto que define como um modelo de linguagem se conecta a ferramentas, dados e prompts de forma padronizada. Antes dele, cada integração era ad-hoc — você escrevia código específico para cada combinação de LLM + ferramenta. Com MCP, qualquer host (Claude Desktop, Cursor, IDE X) fala com qualquer server (GitHub, Postgres, Google Drive) sem retrabalho.

Analogia: assim como o USB-C padronizou a conexão física entre dispositivos e acessórios, o MCP padroniza a conexão lógica entre LLMs e ferramentas. Um cabo, várias fontes de dados e ações.

Arquitetura

[ MCP Host ] ← Claude Desktop, Cursor, Continue, sua IDE │ │ JSON-RPC ▼ [ MCP Client ] ← gerencia 1..N conexões │ ├──▶ [ MCP Server A ] ← GitHub (PRs, issues, files) │ ├──▶ [ MCP Server B ] ← Postgres (queries, schema) │ └──▶ [ MCP Server C ] ← Google Drive (search, read) Transport: stdio · HTTP+SSE · streamable HTTP

Componentes

H

Host

App que conversa com o usuário. Embuti o(s) client(s) MCP. Decide quando e como chamar cada server.

Exemplos: Claude Desktop · Cursor · Continue · Cline · Zed
C

Client

Componente dentro do host que gerencia a conexão com um server específico. Implementa o protocolo JSON-RPC.

1 host → N clients → N servers
S

Server

Processo (local ou remoto) que expõe capabilities ao modelo. Stateless ou stateful, conforme a necessidade.

Catálogo: modelcontextprotocol/servers (oficial) + centenas da comunidade

Capabilities

Um server MCP pode expor três tipos de capability:

CapabilityO que éExemplo
Tools Funções que o modelo pode chamar (similar a function calling) create_issue(), run_query(), send_email()
Resources Dados que o modelo pode ler (similar a arquivos anexados) file://repo/src/auth.ts, db://users/schema
Prompts Templates reutilizáveis que o usuário pode invocar /review-pr, /summarize-thread
Sampling Server pede ao LLM do host para gerar texto (server → LLM) Análise que precisa de outro modelo no meio do fluxo

Por que importa

  • Antes do MCP: cada ferramenta precisava de integração ad-hoc por host. 10 hosts × 50 ferramentas = 500 integrações.
  • Com MCP: uma implementação por ferramenta serve para todos os hosts. 50 tools, escritas uma vez.
  • Adoção rápida: OpenAI, Google DeepMind, Replit, Sourcegraph, Zed, Block, Apollo — todos anunciaram suporte em 2025.
  • Ecossistema crescente: servers oficiais para GitHub, Slack, Postgres, Google Drive, Puppeteer, Filesystem. Centenas de servers da comunidade.
  • Open standard: governança aberta, sem vendor lock-in.

Em 2026, MCP é o padrão de facto para conectar LLMs a ferramentas. Não é o único (OpenAI tem seu próprio tool spec, Anthropic tem tool use nativo), mas é o que se consolidou como camada de interoperabilidade cross-provider.

// 07

Harness — A camada que torna o modelo útil

O modelo sozinho é só um completador de texto. O harness dá a ele mãos, olhos, contexto persistente e uma UX específica para um caso de uso real.

O que define um harness: integração profunda com sistema operacional ou aplicações, contexto persistente (arquivos, codebase, histórico), permissões e segurança, UX desenhada para o domínio, e um modelo de interação que vai além do chat.

⌘

Claude Code

Anthropic · fev/2025

Terminal-first agent. Lê e edita arquivos, roda comandos, busca na web, gerencia git. Suporta skills, sub-agents e MCP nativamente. Filosofia: mínima abstração, máximo controle.

  • Modelo: Claude 4 Sonnet / Opus
  • Interface: CLI
  • Skills, sub-agents, MCP, hooks
  • Ideal para: desenvolvimento, devops, scripting
Comando: claude (instalado via npm)
⌥

GitHub Copilot

GitHub/Microsoft · 2021+

Harness dentro do editor (VS Code, JetBrains, Visual Studio, Neovim). Começou como autocomplete; virou chat e depois agent mode (ago/2024). Workspace agent cria PRs, fecha issues, faz review.

  • Modelos: GPT-4o, Claude 3.5/4, Gemini
  • Interface: IDE + chat + agent
  • Adoção: 1.8M+ usuários pagos, 77k+ organizações
  • Ideal para: desenvolvimento integrado com GitHub
Comando: extensão + Copilot Chat
⊞

Microsoft 365 Copilot

Microsoft · 2023+

Harness dentro do ecossistema Microsoft. Integração profunda com Word, Excel, PowerPoint, Outlook, Teams, OneNote. Copilot Studio permite criar copilots customizados para fluxos específicos da empresa.

  • Modelo: GPT-4 class (com opções)
  • Interface: side-panel em cada app M365
  • Agents específicos: Excel Analyst, PowerPoint Designer
  • Ideal para: produtividade office corporativa
Requer licença: Microsoft 365 Copilot
⊕

Microsoft Cowork RECENTE · GA JUN/2026

Microsoft · GA em 16/06/2026

Agente autônomo que vai além do chat: recebe um objetivo, decompõe em passos, executa usando as tools do Microsoft 365 e entrega o resultado final. Construído sobre o Work IQ — camada de inteligência que lê e-mails, reuniões, arquivos e dados organizacionais para dar contexto às ações.

  • Modelos: Claude Opus/Sonnet, GPT 5.5, Sonnet+Opus Advisor pairing
  • Interface: dashboard visual com progresso multi-step
  • Capabilities: geração de imagem, local browser use, custom skills/plugins
  • Aprovações: pausa antes de ações sensíveis (e-mail, calendário, share)
  • Ideal para: workflows longos, pesquisa, planejamento, automação de processos
Casos de uso: "prepara brief da reunião de amanhã" · "analisa pipeline estagnado" · "compara 4000 arquivos entre versões"

Tendência 2026: a fronteira entre agent e harness está se dissolvendo. Harnesses modernos (Claude Code, MS Cowork) são agents — só que com UX, integrações e governança de produto. A diferença que importa é: o harness é o que você compra/usa, o agent é o padrão arquitetural por trás.
